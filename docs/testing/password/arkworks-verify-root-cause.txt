Root cause
----------
The password input builder emits numeric JSON values for inHaystack, all DFA
state arrays, matchStart, and matchLength. normalize_rust_witness_input only
wrapped string scalars and left numeric values unchanged.

circom-prover 0.1.4 rust-witness::json_to_hashmap accepts only arrays and then
keeps only string elements. Numeric arrays therefore became empty vectors and
numeric scalar fields were omitted. salt and passwordCommitment remained valid
because they were already strings. This explains the observed combination:
correct public commitment, successful Groth16 proof construction, and failed
verification. Arkworks proof construction does not itself enforce witness
satisfaction; verification detects the invalid witness-derived proof.

Fix
---
- Convert every supported Circom scalar to a one-element decimal string array.
- Convert every numeric/string array element to a decimal string.
- Reject unsupported JSON value types instead of silently dropping them.
- Add a password-specific Arkworks diagnostic call that verifies the original
  native proof before conversion and reports proof/zkey SHA-256.
- Hash the proof again after UniFFI reconstruction and require an exact match.
- Remove the temporary Rapidsnark fallback; all tests use Arkworks.
